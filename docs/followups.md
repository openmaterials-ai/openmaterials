## Validation engine (landed 2026-05-20)

`compute` / `compose_executable` / `cross_check` ship in
`omai/representation/{executor,validation}.py` and `omai/operator/compose.py`.
The executor's Sum evaluator now handles elementwise summands and tensor
contractions (einsum); `contract_kappa_direct` is executable. Example A
(molar Cv vs phonopy, 0.1%) and Example B (κ_LBTE vs kaldo, 3.9e-7) pass.

Open follow-ups surfaced during implementation:

- **κ-contraction unit reconciliation.** Contracting canonical-unit inputs
  (c [J/K], v [Å·THz], F [Å], V_cell [Å³]) through `contract_kappa_direct`
  yields κ in J·THz/(K·Å) = 1e22 × W/(m·K). The declared canonical unit of
  `ThermalConductivity` is W/(m·K) (to_operator=1.0), so there is a 1e22
  mismatch between the contraction result and the declared output unit.
  Example B applies the ×1e22 explicitly in the example/test as a documented
  interim. The proper fix is in the executor's contraction layer: derive the
  output's canonical scale from the inputs' canonical units and reconcile it
  against the output space's declared unit, emitting the factor automatically.
  Not a `units.py` value change — the individual unit definitions are correct.

- **Cross-driver `group_velocities_AT.npy` layout split.** `run_kaldo.py`
  (silicon) now saves group velocity as (3, n_q, n_modes) to match the
  GroupVelocity operator field signature (alpha, q, nu); silicon `run_phonopy.py`
  and both germanium drivers still save native (n_q, n_modes, 3). No current
  code path crosses these layouts, but it is a latent trap. Normalize the
  layout — preferably at consume/represent time rather than at each driver's
  save site.

- **`compute_dos` is executable-by-default but cannot lambdify.** Its summand
  contains a DiracDelta δ(ω−ω[q,ν]); the default executability heuristic passes
  it (Eq with disjoint LHS/RHS symbols) but `apply_edge` would hit a
  DiracDelta at lambdify time. No test currently runs it through the executor.
  Either set `is_executable_in_sympy_override=False` on `compute_dos`, or have
  the executor detect non-lambdifiable kernels and raise a clean error.

- **Automatic route enumeration** (cross_check takes routes explicitly),
  **multi-output edge derivation** (compute requires multi-output spaces as
  sources), **cross-mesh route comparison** (examples hold the q-mesh fixed),
  and **live-invoking Source thunks** (codes are loaded oracles today) remain
  open by design.

## Phase 2 — MD-based κ paths (the next major phase)

The user-direction in this consolidation pass: LAMMPS + ASE as Potential
reference; GPUMD as the κ marquee reference (HNEMD / Green-Kubo); a new
class of operator states for MD primitives (HeatCurrent, MSD, VAF,
Trajectory) and new transport-model values on κ
(`green_kubo`, `nemd`, `hnemd`). Decomposed into three sub-projects:

- **P1**: LAMMPS+ASE as Potential / Forces / FC-derivative adapter.
  Anchor role. Specs for the FC2/FC3 chain through ASE's calculator
  protocol. The simplest of the three.
- **P2**: MD primitives — new operator states for time-series outputs
  (Trajectory, HeatCurrent, VelocityAutocorrelation,
  MeanSquaredDisplacement) and edges that produce them.
- **P3**: MD-based κ — `ThermalConductivity[transport_model=*]` for
  Green-Kubo / NEMD / HNEMD; new contraction edges; LAMMPS and GPUMD
  adapter specs cover them.

Brainstorm and spec for phase 2 are deferred; spec_demo will gain MD
sections once the relevant operator-layer nodes exist.

## Operator/representation framework — followup items from the consolidation review

These came out of the spec / code reviews during Task 3 (sub-tasks
3A–3E). Each is well-scoped:

- **Layering smell in `validate.py`**. The new `_STATE_SYMBOLS` registry
  (per-state allowed sympy symbol names) lives in
  `omai/operator/validate.py` but contains thermal-transport-specific
  LaTeX strings (`\kappa^{W,coh}`, `\Gamma^{bnd}`,
  `CumulativeKappa[wrt=mfp]`, etc.). The framework module should be
  domain-agnostic. Promote the registry to a public attribute on `State`
  / `Field` (e.g. a `symbol_name` field) or move per-domain registries
  into `omai/thermal_transport/operator/`. The current implementation
  ships, but a non-thermal-transport domain would either pollute or
  duplicate.

- **Drift risk between `_STATE_SYMBOLS` and the actual IndexedBase
  declarations.** Symbol names are duplicated as string literals in
  `validate.py` while the canonical definitions are
  `sp.IndexedBase(...)` instances in `omai/thermal_transport/operator/
  edges.py`. A rename in `edges.py` silently bypasses validation.
  Expose the IndexedBase objects from `edges.py` and read `.name` in
  `validate.py`.

- **Private-name import in `executor.py`**:
  `from omai.operator.validate import _STATE_SYMBOLS`. Underscore name
  is by convention private; the executor reaching into the validator's
  internals is a coupling smell. Either rename to public
  (`STATE_SYMBOLS`) or wrap in a helper (`state_primary_symbols(state)`).

- **Hardcoded ℏ_eff = h·1e12 (linear-THz) duplicated** between
  `omai/representation/executor.py` and `tests/test_executor.py`. Hoist
  into `omai/operator/constants.py` or a similarly placed module.

- **`executor.py` size (528 lines).** On the edge of single-responsibility.
  Could split: `apply_edge.py` (the entry point + dispatch),
  `lambdify_helpers.py` (the n_BE expansion, Sum elimination,
  Indexed→dummy bindings), `operator_form.py`
  (`_OperatorFormStateAdapterSpec`, `operator_form_spec`). Not
  blocking; cost is bounded.

- **Silent-pass on unregistered new States in `validate_dag`'s
  free-symbol check.** `_STATE_SYMBOLS.get(name, frozenset())` returns
  empty for unknown states, which means a new State's formula could
  pass with zero coverage if its symbols happen to all be permitted
  constants. Make the empty-registry case a soft warning or violation.

- **Test coverage for the sympy validators is partial.** The unit tests
  added in Task 3C cover the free-symbol check, the LHS-index check,
  and the auxiliary-formula check. Tests for `Symbol`-typed LHS (not
  Indexed), for the multi-output edge case, and for the 3+ edge
  composition chain in `compose.py` would harden coverage.

- **`compose.py` substitution semantics — strengthen the test.** The
  current intermediate-IndexedBase-removed test asserts only that the
  `c` symbol is gone after `compute_heat_capacity → contract_molar_*`.
  Could additionally check the substituted form contains
  the expected ω-sinh structure inside the Sum.

- **Adversarial branch of `compare_operators` in
  `tests/test_op_rep_boundary.py` is structurally unreachable.** All
  106 cross-adapter pairs land on `same_operator=True`. The assertion
  code handles the False branch correctly but no real data triggers
  it. Add a synthetic `compare_operators` case with deliberately
  mismatched specs.

- **`KALDO_HELMHOLTZ_FREE_ENERGY` declares unit `eV`** but `eV` isn't
  in `omai/representation/units.py`'s `UNITS` registry. The round-trip
  test skips for that case. Either register `eV` + `J` for the
  `ENERGY` dimension, or remove the unit declaration. Real gap.

- **Discovery of adapter specs is duplicated** between
  `omai/thermal_transport/visualize.py::_collect_specs` and
  `tests/test_op_rep_boundary.py`. Expose a shared helper in the
  package (e.g. `omai.thermal_transport.representation.list_specs()`).

## Representation layer

- **The FC3 0.1 factor (resolved 2026-05-17).** Traced to a hidden
  nm/Å unit mix in ShengBTE: `shengbte/Src/gruneisen.f90:44`
  documents the FC3-related unit chain as
  `nm·eV/(amu·Å³·THz²)` — nm in the numerator with Å³ in the
  denominator, an implicit 10× factor (1 nm = 10 Å). For the same
  physical Φ³, ShengBTE expects values 0.1× the natural eV/Å³ form.
  Now encoded as the `fc3_normalization` convention on
  `ForceConstants[order=3]` (canonical = `eV_per_A3`, ShengBTE override
  = `eV_per_A2_per_nm` with convention_factor 0.1).
  `inter_representation_factor(phono3py_or_kaldo_spec,
  shengbte_spec, "phi")` returns 0.1 mechanically.
  `experiments/silicon_shengbte/convert.py` now derives the factor
  from the framework instead of hardcoding it. Open kaldo PR (closed)
  for `save_third_order_matrix`: `gb/fix-shengbte-fc3-factor` branch.

- **kaldo's `Phonons.free_energy` is BZ-averaged** (divided internally
  by `n_k_points`), while the operator layer's canonical
  `HelmholtzFreeEnergy` is per-mode. Documented in the kaldo spec note
  but cross-code comparison at this Observable currently requires
  manually accounting for the difference. Either rename the kaldo spec
  to target `MolarHelmholtzFreeEnergy` (after the contraction) or
  re-derive a per-mode form from kaldo internals.
